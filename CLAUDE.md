# CLAUDE.md ‚Äî dsa-content

A **content repo** (not an app) for the `graphl-ux` engine. It holds the **Data
Structures & Algorithms** concept: notebooks, per-section narration, and the
`manifest.json` that wires them. The app fetches this repo's `manifest.json` +
notebooks over the network and renders them. Read `README.md` first; this file is the
working orientation.

This repo is a sibling of `apache-spark-content`, `java-content`, `scala-content`, and
`python-content` and follows the **same contract** ‚Äî when in doubt, mirror what
`apache-spark-content` does (its `CLAUDE.md`/`DESIGN.md`/`MODULES.md` are the mature
reference, especially the **TTS guidelines** and the **fixed semantic palette**).

## The core contract (do not break)

1. **The notebook is the single source of truth** for prose and code. `manifest.json`
   only *wires* sections ‚Äî never duplicate notebook content here.
2. The app splits each notebook at every `## ` heading into sections (= pages) and
   matches them to the manifest by **normalized heading** (lowercase, backticks
   stripped, whitespace collapsed). A heading edit in a notebook must be mirrored in
   the manifest `heading`.
3. Section diagram **images are stripped** by the app ‚Äî a **scene** replaces them.
4. **Scenes live in `graphl-ux`** (`src/scenes`), authored in TypeScript ‚Äî not here.
   The `scenes/` dir is reserved/empty. Reference a scene by **id** only.

## The DSA scene (in graphl-ux, not here)

DSA rides **one** scene, **`dsa`** (`graphl-ux/src/scenes/dsa.ts`, registered in
`scenes/index.ts`): the "top data structures" poster laid out GROUPED BY taxonomy
category ‚Äî the grouping IS the categorisation, and each cell shows what the structure
looks like (boxed sequences + node-link diagrams). (History: this replaced an earlier
two-column **Taxonomy ‚ñ∏ RAM** map; the RAM sketches were dropped, and a brief two-scene
split ‚Äî `dsa` + `dsa-catalog` ‚Äî was merged back into this single grouped scene. **No
`dsa-ram-*` ids exist anymore**, and there is no separate `dsa-catalog` scene.)

- **Category groups** (titled bands): `dsa-grp-primitive`, `dsa-grp-static`,
  `dsa-grp-dynamic`, `dsa-grp-tree`, `dsa-grp-graph`, `dsa-grp-hashing`.
- **Structure boxes** (container ids): `dsa-cat-array`, `dsa-cat-matrix`,
  `dsa-cat-queue`, `dsa-cat-stack`, `dsa-cat-linkedlist`, `dsa-cat-tree`, `dsa-cat-bst`,
  `dsa-cat-heap`, `dsa-cat-trie`, `dsa-cat-graph`, `dsa-cat-unionfind`, `dsa-cat-hashmap`
  (plus fine sub-ids like `dsa-cat-bst-4`, `dsa-cat-ll-n0`). The `dsa-grp-primitive`
  band holds the primitive chips (`dsa-cat-prim-int`, ‚Ä¶).

Highlighting a container id lights its children, so spotlight a group (`dsa-grp-tree`)
or a box (`dsa-cat-array`) to light the whole thing. Always check `dsa.ts` for the id
list before adding a `highlight`/`focus`.

## manifest.json shape

Top level: `concept`, `design`, `scenes[]` (just `dsa`), `presentations[]` (one per
module). Each presentation: `id`, `title`, `notebook`, `defaultScene` (always `"dsa"`),
`sections[]`. Each section overlay: `heading` (must match a notebook `## ` heading,
normalized), `scene` (always `"dsa"`), `spine` (bool ‚Äî feed-mode narration; `false` for
recap/cheat-sheet/summary sections), optional `role` (`"hook"` on the first section),
`highlight` (scene node ids ‚Äî AMBER, rest dim), `focus` (node id(s) the camera frames;
both **omitted** for module 01's overview sections, which keep the whole poster framed),
`audio` (repo-relative `.wav`).

`manifest.json` is generated by `scripts/build_manifest.py`: per section it picks the
box/category the camera frames (`highlight` + `focus`) by topic keyword (`KEYWORD`),
falling back to a per-module default (`MODULE_DEFAULT`; `None` = whole poster). Re-run it
after any notebook heading change.

## Narration

One `.tts` per section, plain spoken prose. Naming `tts/<NN>-<SS>-<slug>.tts` ‚Üí
`audio/<NN>-<SS>-<slug>.wav`; the stem is shared by the `.tts`, the `.wav`, and the
manifest `audio` field, where `<slug>` is the normalized notebook heading. **Drop the
notebook's intro overview and the trailing "next notebook" outro** ‚Äî those are
reading-only, not narrated. Follow `apache-spark-content/CLAUDE.md` "TTS guidelines"
verbatim (no markdown/code; spell out symbols/acronyms ‚Äî `O(n)` ‚Üí "linear time",
RAM ‚Üí "ram", `==` ‚Üí "double equals", `heapq` ‚Üí "heap-q", `O(h)` ‚Üí "proportional to the
height").

The source curriculum at `~/Projects/dsa/tts/` has **one `.tts` per notebook** (a
monologue). The per-section files here were authored 1:1 with the notebook `## `
headings, reusing the monologue prose where it aligns and filling sections the
monologue narrated only coarsely (the curriculum monologue is coarser-grained than the
notebook sections, so it could not be split mechanically).

## When wiring a module

1. Confirm the notebook is in `notebooks/` and its `## ` headings are final.
2. Author one `tts/<NN>-<SS>-<slug>.tts` per section (1:1 with the headings).
3. Run `scripts/build_manifest.py` to regenerate `manifest.json` (scene `dsa` +
   spine/role/highlight/focus/audio). A new module needs a `MODULE_DEFAULT` entry (and
   any new section keywords) in that script.
4. Run `scripts/colab_generate_audio.ipynb` to generate + push the `.wav`s.
